# 安全策略

## 基本原则

- 不默认删除未知文件。
- 不默认迁移系统关键目录。
- 不默认处理账号、密钥、授权数据。
- 清理前展示明细。
- 迁移前进行检测。
- 操作后记录日志。
- 可回滚的操作必须提供回滚入口。

## 风险等级

| 等级 | 处理原则 | 示例 |
|---|---|---|
| safe | 可推荐清理 | 临时文件、回收站、普通缓存 |
| medium | 用户确认后处理 | 用户目录迁移、开发工具缓存、Junction |
| high | 默认只扫描提示 | 系统目录、凭据目录、Docker、WSL、虚拟机 |

## 清理策略

清理前必须扫描并统计文件数量、预计释放空间、风险等级和具体路径。清理时必须跳过被占用文件、权限不足文件和系统保护文件。

清理能力分阶段开放：

| 阶段 | 能力 | 是否删除文件 |
|---|---|---|
| 预览阶段 | 统计预计释放空间、文件数量、跳过项、路径明细 | 否 |
| 确认阶段 | 展示风险等级和二次确认要求 | 否 |
| 执行阶段 | 按规则删除或清空，并写入日志 | 是 |

执行阶段上线前，清理命令必须保持 `executable=false`，前端不得显示真实执行按钮。

确认阶段可以记录清理计划草稿。草稿状态必须为 `planned`，只能作为审计和后续确认依据，不能代表已经执行清理。

第一版真实清理只允许使用隔离移动策略：

- 仅开放 `temp-user` 规则。
- 只处理 `%TEMP%` 的直接子项。
- 文件和目录移动到 `.clearc/quarantine/<operation-id>/`。
- 不永久删除文件。
- 不清空回收站。
- 被占用、无权限或匹配排除规则的项目必须跳过并记录。
- 操作日志必须标记为可回滚。

隔离清理回滚策略：

- 回滚只能处理 `cleanup-quarantine` 操作。
- 回滚必须读取日志中的 `movedItems`，逐项恢复到原路径。
- 如果原路径已存在，必须跳过并记录失败原因。
- 如果隔离文件不存在，必须跳过并记录失败原因。
- 每个隔离清理操作只能产生一条回滚记录。
- 回滚本身必须写入操作日志，且不可再次回滚。

## 迁移策略

迁移前必须检查目标盘是否存在、空间是否足够、目标路径是否可写、是否为移动设备、原路径是否被占用、目标目录是否冲突。

迁移过程需要保留原路径、新路径、修改前配置、修改后配置、操作时间、文件数量和总大小。

用户目录迁移分阶段开放：

| 阶段 | 能力 | 是否修改注册表或移动文件 |
|---|---|---|
| 只读检测 | 读取 Shell Folder 当前配置，展示路径状态 | 否 |
| 预检查 | 检查目标盘、权限、冲突和空间 | 否 |
| 执行迁移 | 移动文件并修改 Shell Folder | 是 |

在执行迁移上线前，迁移模块不得修改注册表、不得移动用户目录内容。

预检查阶段可以计算目标路径、读取磁盘容量、判断路径是否存在，但不得通过创建临时文件验证可写性；可写性必须标记为 `not-verified`，留到执行阶段确认。

用户目录执行迁移策略：

- 执行迁移必须先通过预检查。
- 执行迁移必须由用户点击触发，并带确认短语。
- 迁移只移动源目录的直接子项，不删除源目录本身。
- 目标路径已存在时不得覆盖。
- 跨盘迁移不得只依赖系统重命名；重命名失败时必须使用复制、大小校验、删除源文件的受控流程。
- 如果复制后删除源文件失败，必须尝试清理已复制目标，且必须记录失败原因。
- 修改 `User Shell Folders` 前必须记录原注册表值和原注册表值类型。
- 写入 `User Shell Folders` 时应尽量保留原值类型；原值不存在时默认使用可展开字符串类型。
- 注册表写入失败也必须写入迁移日志，不能让已移动文件缺少审计记录。
- 操作日志必须记录 `movedItems`、原路径、目标路径、原注册表值、新注册表值、注册表写入状态和失败项。
- 回滚必须恢复原注册表值，并逐项恢复文件。
- 回滚文件恢复必须复用受控移动流程，避免跨盘回滚失败。
- 注册表恢复失败也必须写入回滚日志。
- 同一个迁移操作只能回滚一次。

## 开发工具管理策略

开发工具管理第一阶段只允许只读扫描：

- 可以读取规则路径。
- 可以统计存在性、文件数、目录大小和跳过项。
- 不得清理开发工具目录。
- 除 `V2 / M11.1` 明确支持的 Junction 执行流程外，不得移动开发工具目录。
- 除 `V2 / M11.1` 明确支持的 Junction 执行流程外，不得创建 Junction。
- 不得修改环境变量。
- `.aws` 等凭据目录只能展示为高风险只读项。

开发工具环境变量迁移策略：

- 仅允许规则中 `preferredMove=env` 且包含 `env` 配置的工具进入环境变量迁移流程。
- 当前允许 `ANDROID_USER_HOME`、`CARGO_HOME`、`RUSTUP_HOME`。
- `.aws`、凭据目录、AI 工具授权目录和高风险目录不得自动迁移。
- 执行前必须生成计划，展示原环境变量值、新环境变量值和目标路径。
- 执行前必须由用户输入确认短语并通过确认弹窗。
- 执行阶段只允许创建目标目录和修改用户级环境变量，不自动移动已有开发工具文件。
- 操作日志必须记录原值、新值、目标路径、失败项和回滚信息。
- 回滚必须恢复原环境变量值；如果原值不存在，回滚时必须删除新增变量。
- 环境变量变更后需要提示用户重启终端或相关开发工具。

开发工具 Junction 预案策略：

- 仅允许规则中 `preferredMove=junction` 或 `fallback=junction` 的工具生成 Junction 预案。
- 高风险目录不得显示可用的 Junction 预案按钮。
- `M11` 只允许检测和生成计划，不移动文件、不创建 Junction。
- 预案必须展示源路径、目标路径、源路径大小、目标盘状态、目标剩余空间、目标路径冲突和源路径是否已是 Junction。
- 推荐只读的规则可以生成评估预案，但必须提示“仅用于评估，不开放执行”。
- 真实创建 Junction 在 `V2 / M11.1` 实现，必须增加确认短语、二次确认、文件迁移日志和回滚策略。
- 删除 Junction 前必须确认原路径是本应用创建的 Junction，不得递归删除真实目标目录。

## 软件迁移策略

V1/V2 不自动迁移已安装软件目录。优先级如下：

1. 使用软件官方设置修改路径。
2. 使用环境变量修改路径。
3. 使用配置文件修改路径。
4. 使用 Junction。
5. 只提示用户，不自动处理。
