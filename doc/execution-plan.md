本文件用于指导实际开发。版本目标参考 [版本路线图](./roadmap.md)，安全边界参考 [安全策略](./safety-policy.md)。执行时必须同步当前版本、里程碑、验证结果和相关文档。

## 当前执行目标

当前进入 `V2：可控迁移能力` 的收口核对阶段。`V0`、`V1`、`V1.1` 和 `V2` 已知编码项均已实现；涉及真实文件迁移、注册表、环境变量和 Junction 的能力需要后续在受控测试环境中手动验收。

- `V0`：已完成工程初始化、页面框架、规则系统、Native Core 基础结构。
- `V1`：已完成只读扫描、安全清理、日志回滚、开发工具只读扫描和发布准备。
- `V1.1`：已完成可用性修复，长任务已后台化并补充 loading / 禁用原因。
- `V2`：代码实现已完成。`M8-M9` 用户目录迁移执行与回滚、`M10` 开发工具环境变量迁移、`M11.1` Junction 真实执行与回滚均进入后续真实手动验收；`M11` Junction 迁移预案和 `M12` 任务体验增强已完成。

## 执行同步约定

每次执行开发任务后，都需要同步：

- 当前执行到哪个版本、里程碑和步骤，例如 `V2 / M10 / Cargo 环境变量迁移`。
- 本次完成了哪些改动。
- 本次是否修改了系统行为、安全策略、规则结构或页面流程。
- 本次验证了哪些命令，验证结果是什么。
- 是否存在需要用户确认的问题。
- 是否已经同步相关文档。
- 如果变更影响用户可见功能、按钮流程、结果展示、错误提示或安全提示，必须同步 `user-manual.md`。

如果实现内容影响产品范围、版本计划、安全边界、架构或开发流程，必须同步更新对应文档：

| 影响范围 | 需要同步 |
|---|---|
| 产品功能和边界 | `product-plan.md`、`roadmap.md` |
| 版本和里程碑 | `roadmap.md`、`execution-plan.md` |
| 删除、迁移、权限、回滚 | `safety-policy.md` |
| 开发工具和 AI 工具迁移 | `devspace-manager.md` |

| 打包发布 | `release.md` |

## V1：安全清理 MVP

### M2：扫描系统

验收：

- Rust 能解析 `%USERPROFILE%`、`%TEMP%`、`$Recycle.Bin` 等规则路径。
- Rust 能读取系统盘总容量、可用容量和已用容量。
- Rust 能对规则命中的路径做只读递归统计。
- 扫描结果包含存在性、可访问性、文件数量、目录数量、占用大小和跳过数量。
- Dashboard 页面展示系统盘概览。
- Analysis 页面展示规则扫描结果。
- 扫描阶段不执行删除、迁移或配置修改。

### M3：安全清理

验收：

- 清理预览能基于清理规则生成预计释放空间。
- 清理预览能展示文件数量、跳过数量、路径明细和风险等级。
- 清理预览默认要求二次确认。
- Cleanup 页面可以记录清理计划草稿。
- 第一版真实清理只允许 `temp-user` 隔离移动，不允许永久删除。
- 隔离清理结果必须记录移动数量、跳过数量、隔离目录和失败原因。

### M4：日志回滚

验收：

- 隔离清理日志记录 `movedItems`，包含原路径和隔离路径。
- Logs 页面能识别可回滚的 `cleanup-quarantine` 记录。
- 回滚命令能把隔离文件恢复到原路径。
- 原路径已存在或隔离文件缺失时，回滚必须跳过并记录失败。
- 同一个隔离清理操作不能重复回滚。
- 回滚操作会写入新的日志记录。

### M5：用户目录只读检测和预检查

验收：

- 迁移模块能读取 Windows `User Shell Folders` 当前配置。
- 支持检测桌面、下载、文档、图片、视频、音乐。
- 页面展示配置路径、解析后路径、是否存在、是否仍在系统盘。
- 迁移预检查能评估目标盘是否存在、目标目录是否冲突、目标是否仍在系统盘、目标空间是否足够。
- 预检查阶段不创建目录、不写文件、不修改注册表。

### M6：开发工具只读扫描

验收：

- DevSpace 规则覆盖 Android、Cargo、Bun、通用缓存、Chromium snapshots、AI 工具、Apifox、CodeBuddy、AWS。
- DevSpace 扫描能统计路径存在性、文件数量、占用空间和跳过项。
- DevSpace 页面展示风险等级、分类、推荐动作和迁移方式。
- 当前阶段只读扫描，不清理、不迁移、不修改环境变量、不创建 Junction。

### M7：打包发布准备

验收：

- Tauri 打包配置包含 Windows 安装包目标。
- `package.json` 提供桌面开发和生产打包脚本。
- 发布文档说明安装包、MSI 和便携版策略。
- 发布前检查项覆盖构建、格式、安全边界和文档同步。

## V2：可控迁移能力

### M8：用户目录迁移执行

当前状态：

- 代码加固已完成。
- 已打通前端入口、Rust command、迁移计划、执行结果和操作日志。
- 已补充跨盘移动 fallback、注册表值类型保留、失败日志和前端确认短语。
- 尚未对当前机器真实用户目录执行迁移验收。

已完成：

- Migration 页面支持生成迁移计划。
- `create_user_folder_migration_plan` 已接入预检查。
- `execute_user_folder_migration` 已接入确认短语、目标目录创建、直接子项移动、`User Shell Folders` 写入和日志记录。
- 文件移动会先尝试系统重命名；失败时使用复制、大小校验、删除源文件的受控 fallback。
- 注册表写入会尽量保留原值类型，默认使用 `REG_EXPAND_SZ`。
- 注册表写入失败会进入迁移日志，不再让文件已移动但无审计记录。
- 前端执行前需要输入 `MIGRATE_USER_FOLDER`，并通过确认弹窗展示源路径和目标路径。
- 执行结果可以展示移动项和失败项。
- 迁移日志记录原注册表值、目标注册表值、原路径、目标路径、移动项和失败项。

后续执行/手动验收：

- 需要补充真实迁移手动验收记录。开发和自动测试阶段不得迁移当前机器真实用户目录。
- 需要执行后校验，确认注册表、目标目录、移动项和失败项状态一致。

后续增强：

- 可补充 Explorer 刷新或更明显的重新登录/重启资源管理器提示，否则系统默认目录显示可能延迟更新。
- 可补充更完整的集成测试，包括日志字段完整性、注册表写入失败、跨盘复制失败。

验收：

- Migration 页面可以基于预检查生成用户目录迁移计划。
- 执行迁移前必须通过预检查，并要求确认短语。
- 执行迁移会创建目标目录，移动源目录直接子项，并写入 `User Shell Folders`。
- 迁移日志必须记录原注册表值、目标注册表值、原路径、目标路径、移动项和失败项。
- 迁移失败或部分失败时，不删除原目录、不覆盖目标已有文件。

### M9：用户目录迁移回滚

当前状态：

- 代码加固已完成。
- 已打通 Logs 页面入口、Rust command、回滚日志和重复回滚拒绝。
- 已补充跨盘回滚 fallback、注册表恢复失败日志和回滚确认弹窗。
- 尚未对当前机器真实用户目录执行回滚验收。

已完成：

- Logs 页面能识别 `user-folder-migration` 日志并调用迁移回滚命令。
- `rollback_user_folder_migration` 能根据迁移日志恢复文件项和注册表原值。
- 回滚文件恢复会复用受控移动流程，支持跨盘复制、校验、删除源文件。
- 回滚会跳过原路径冲突、目标文件缺失等失败项，并记录结果。
- 注册表恢复失败会进入回滚日志，并展示 `rolled-back-with-registry-failure` 状态。
- Logs 页面回滚前会弹出确认提示，并区分“回滚迁移”和“回滚清理”。
- 同一个迁移操作重复回滚会被拒绝。
- 回滚操作会写入新的不可回滚日志。

后续执行/手动验收：

- 需要补充真实回滚手动验收记录。开发和自动测试阶段不得回滚当前机器真实用户目录。

后续增强：

- 可把可回滚日志列表做完整展示、筛选和详情页，当前页面优先展示最近操作。
- 可补充缺失日志、日志字段缺失、原注册表值为空等测试。

验收：

- Logs 页面能识别可回滚的 `user-folder-migration` 记录。
- 回滚迁移会逐项把文件从目标路径恢复到原路径。
- 回滚会恢复 `User Shell Folders` 原值。
- 原路径已存在或目标文件缺失时，回滚必须跳过并记录失败。
- 同一个迁移操作不能重复回滚。
- 回滚操作会写入新的不可回滚日志记录。

### M11.1：Junction 真实执行与回滚

当前状态：

- 代码实现已完成，待手动验收。
- 依赖 `M11` 的 Junction 迁移预案结果。
- 属于 `M11` 的执行延伸和 V2 可控迁移能力的后续收口，不放到 V3 深度分析。
- 已接入 DevSpace 页面执行入口、Logs 页面回滚入口、Rust command 和可回滚日志。

目标：

- 对通过预案的开发工具目录执行真实迁移。
- 将源目录内容移动到目标路径。
- 在原路径创建 Junction。
- 写入可回滚日志。
- 支持 Junction 回滚。

执行边界：

- 只允许 `preferredMove=junction` 或 `fallback=junction` 的规则进入执行。
- 高风险目录、凭据目录、Docker、WSL、虚拟机、数据库类目录不开放执行。
- 目标路径已存在时必须阻塞，不覆盖。
- 源路径已经是 Junction、符号链接或异常重解析点时必须阻塞或只读提示。
- 执行前必须输入确认短语，并二次确认源路径和目标路径。
- 创建 Junction 前必须完成文件移动、目标校验和日志草稿记录。
- 回滚时必须先确认原路径是本应用创建的 Junction，再删除 Junction 并恢复文件。
- 删除 Junction 时必须区分 Junction 本身和真实目标目录，禁止递归删除目标目录。

验收：

- 能基于 M11 预案生成可执行计划。
- 目标冲突、源路径异常、空间不足会阻塞执行。
- 执行后原路径为 Junction，目标路径包含迁移内容。
- 日志记录原路径、目标路径、移动项、Junction 状态和失败项。
- 回滚能删除 Junction 并把文件恢复到原路径。
- 重复回滚被拒绝。
- 失败和取消时不会覆盖目标已有文件，不会删除源目录中未确认处理的内容。
